# code-review

Use before opening a PR or committing a larger change. Skip for single-line typo fixes.

## When NOT to use
- One-line doc fix
- Revert of a known-bad commit
- Autogenerated code (lockfiles, snapshots) — review the config that generated it instead

## Pass order

Go through the change **in this exact order**. If any phase fails, stop and fix before the next.

### 1. Security

- Any new user input? Validated and sanitized?
- Any secrets / tokens / keys in the diff? (grep for `pk_`, `sk_`, `api`, `token`, `password`)
- Any shell command built from user input? (command injection)
- Any SQL built from user input? (use parameterized queries)
- Any file path built from user input? (path traversal)
- Any new dependency? (typosquat check, maintainer history, license)

### 2. Correctness

- Does every changed function still do what its name and docstring say?
- Are edge cases covered? (empty, null, boundary, concurrent)
- Partial-failure handling: does the code surface skipped records / truncated output / retry exhaustion?
- Do the tests actually verify behavior, or just shape?

### 3. Convention

- Does the code match the surrounding style? (indentation, naming, error handling pattern)
- Any new pattern introduced that conflicts with an existing pattern elsewhere?
- Logging / metrics: named consistently with existing loggers?

### 4. Blast radius

- What else could this change affect? (callers, subscribers, scheduled jobs)
- Reversible? If it ships broken, how do you roll back?
- Any migrations, schema changes, or feature-flag flips that can't be undone cleanly?

## Pitfalls

- **Starting with style.** Security first. Style last. If you only have 5 minutes, skip style.
- **Rubber-stamping AI diffs.** AI-written code often fails phase 1 (SQL injection, path traversal) and phase 2 (silent partial failure). Read every line.
- **Skipping blast radius.** The bug that hurts most is the one in an unrelated system the reviewer didn't think to check.

## Verification

- [ ] All 4 phases completed in order
- [ ] No secrets in the diff (confirmed with grep, not just eyeballing)
- [ ] Tests exist for the new behavior and they assert on behavior
- [ ] Rollback plan written if the change is non-reversible
